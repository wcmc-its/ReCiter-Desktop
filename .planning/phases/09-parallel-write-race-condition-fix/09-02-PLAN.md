<objective>
Wave 1: Replace the three SELECT+INSERT/UPDATE loops in `_process_one_researcher` with atomic `INSERT ... ON DUPLICATE KEY UPDATE` statements.

Purpose: This is the core fix for DB-01. The current `db.query()` + `db.add()` interleaving pattern is non-atomic across concurrent ThreadPoolExecutor workers, causing SQLAlchemy autoflush IntegrityErrors when two workers write the same PMID and StaleDataErrors on score re-runs. The upsert pattern eliminates both failure modes at the MariaDB level with a single atomic statement per table.

Output: Modified `api/services/pipeline_runner.py` with 3 upsert replacements and 2 new imports. All tests green.
</objective>

<threat_model>
## Threat Model (ASVS Level 1)

**Scope:** Production code change in `api/services/pipeline_runner.py` — database write path.

| Threat | Severity | Mitigation |
|--------|----------|------------|
| SQL injection via upsert values | LOW | All values pass through SQLAlchemy parameterized bindings (`:param` style). No string concatenation or f-strings in SQL. The `text('NOW()')` is a literal server function call, not user input. |
| Data loss on upsert — silently overwriting article data | MEDIUM | Article upsert uses no-op `on_duplicate_key_update(pmid=stmt.inserted.pmid)` which preserves all existing article fields. PersonArticle upsert is also no-op for existing rows. Only PersonArticleScore intentionally overwrites scores on re-run (correct behavior). |
| `created_at` timestamp corruption | LOW | `created_at` is excluded from all three `on_duplicate_key_update()` clauses. Server default fires only on INSERT. Verified in RESEARCH.md pitfall 3. |
| Empty values list producing invalid SQL | LOW | All three upserts are guarded with `if rows:` before `db.execute()`. Verified in RESEARCH.md pitfall 1. |
| Session identity map staleness after execute() | LOW | Each worker creates a fresh `SessionLocal()` per call. No pre-loaded identity map exists at upsert time. Verified in RESEARCH.md pitfall 4. |

**Blocking threats:** None. The MEDIUM-severity data loss threat is fully mitigated by the no-op upsert pattern.
</threat_model>
