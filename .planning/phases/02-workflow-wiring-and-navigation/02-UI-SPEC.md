## Surface Specifications

### 1. Sidebar — "Statistics" Item

Add one entry to the `workflowItems` array in `frontend/components/sidebar.tsx`.

**Position:** After "Results" (index 5 in the array), before the "Help" section.

**Shape:** Matches existing `workflowItems` entry exactly:
```
{ href: "/stats", label: "Statistics", status: hasAssertions ? "complete" | "next" : "locked" }
```

**Status logic:**
- `locked` when `assertionCount === 0` (or null/loading) — renders `text-gray-500`, icon `—` in `text-gray-600`
- `next` when `assertionCount > 0` and not visited — renders `text-[#cf4520]` dot icon
- `complete` is not applicable for Phase 2; use `next` as the unlocked state (stats page is a view, not a step to complete)

**Exact class string when locked:**
```
text-gray-500 hover:bg-[#2c3e50]/30
```

**Exact class string when unlocked (not active):**
```
text-gray-300 hover:text-white hover:bg-[#2c3e50]/60
```

**Exact class string when active:**
```
bg-[#2c3e50] text-white font-semibold
```

Note: Gate enforcement happens at /stats via PrerequisiteGate, NOT via `pointer-events-none` on the sidebar link. (Source: CONTEXT.md D-04)

### 2. Pipeline Completion — "View Statistics" CTA

**Location:** Inside the `{pipelineFinished && summary && (...)}` block, in the "Next steps" `<div className="flex gap-3">` at the bottom of the completion section.

**Condition:** Render only when `assertionCount > 0`. The link is completely absent (not grayed) when no assertions exist. (Source: CONTEXT.md D-06)

**Placement:** Second button in the flex row, after "View Results":
```
[View Results]  [View Statistics]  [Export All Scores (CSV)]
```

**Button variant:** `variant="default"` with `className="bg-[#cf4520] hover:bg-[#a3381a] text-white"` — same style as the existing "View Results" primary button. This gives the CTA equal visual weight.

**Navigation:** Same tab (`<Link href="/stats">`). (Source: CONTEXT.md — Claude's Discretion; same-tab matches all other internal navigation in the app)

### 3. /stats Route — Gate + Placeholder Page

**File:** `frontend/app/stats/page.tsx`

**Gate condition:** `assertionCount === 0` — gate fires when no joined (score, assertion) pairs exist.

**Gate card copy** (PrerequisiteGate `message` prop):
> "Statistics require scored articles with accepted or rejected decisions. Run the pipeline and import assertions first."

**Gate action button:**
- `actionLabel`: "Go to Pipeline"
- `actionHref`: "/pipeline"

**Gate visual:** Uses existing `PrerequisiteGate` component unchanged — amber card (`border-amber-200 bg-amber-50`), amber button (`border-amber-300 text-amber-700 hover:bg-amber-100`).

**Page body when gate is met (Phase 2 placeholder):**
- Page heading: `text-2xl font-semibold text-gray-900` — "Statistics"
- Subheading: `text-sm text-gray-500` — "Scoring quality metrics for your pipeline run."
- Placeholder notice: a single `Card` with `border-gray-200 bg-white shadow-sm`, containing `text-sm text-gray-500 text-center py-8` — "Charts are coming in Phase 3. Your stats data is ready."
- No loading spinner for Phase 2 placeholder (data fetch is Phase 3 scope). (Source: CONTEXT.md D-08)

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA (pipeline completion) | "View Statistics" |
| Sidebar nav label | "Statistics" |
| Gate message | "Statistics require scored articles with accepted or rejected decisions. Run the pipeline and import assertions first." |
| Gate action button | "Go to Pipeline" |
| Stats page heading | "Statistics" |
| Stats page subheading | "Scoring quality metrics for your pipeline run." |
| Placeholder body (Phase 2) | "Charts are coming in Phase 3. Your stats data is ready." |
| Empty state (gate shown, no assertions) | Gate card — see Gate message above |
| Error state | Not applicable in Phase 2 (no data fetch, no destructive actions) |
| Destructive confirmation | None — no destructive actions in Phase 2 |

Notes:
- "View Statistics" follows the exact verb+noun pattern of the existing "View Results" CTA. Consistent phrasing aids discoverability.
- Gate message avoids technical terms ("joined pairs", "assertion_count") — uses plain language describing the user action required.
- "Go to Pipeline" mirrors the PrerequisiteGate convention in `results/page.tsx` ("Go to Pipeline").
